## 1. 核心模型（PM 必会）

### AARRR（海盗模型）
- **解释**：Acquisition（获客）→ Activation（激活）→ Retention（留存）→ Revenue（变现）→ Referral（推荐），衡量产品增长的核心漏斗
- **案例**：配送机器人上线初期，通过优化"一键下单"流程，激活率从 20% 提升到 55%
- **PM 应用**：每一步都要有明确指标 + 优化动作，不能只盯"获客"不看"留存"

### KANO 模型
- **解释**：基本型需求（必须有）/ 期望型需求（越满足越满意）/ 兴奋型需求（超出预期）
- **PM 应用**：做需求评估时，先分清属于哪一型——基本型必做，期望型性价比最高，兴奋型是差异化

### MVP（最小可行产品）
- **解释**：用最小成本验证核心假设的产品版本
- **PM 应用**：先做"刚好能验证市场"的版本，不要一开始就做"大而全"

### PMF（产品市场匹配）
- **解释**：产品真正满足市场需求，能留住用户
- **PM 应用**：MVP → PMF 是创业产品必经之路，PMF 没找到之前不要盲目扩张

### UCD（用户中心设计）
- **解释**：User-Centered Design，以用户需求为出发点的设计哲学
- **PM 应用**：先访谈真实用户（不是脑补"用户需要什么"），用原型验证再迭代
- **对比**："老板说要加什么功能"或"竞品有什么我们也要有"是反 UCD

### JTBD（用户的待办任务）
- **解释**：Jobs To Be Done，用户"雇用"产品来完成什么任务
- **PM 应用**：用户买的不是"产品"，是"完成任务的能力"——理解任务比理解功能更重要

### OKR vs KPI
- **OKR**：Objectives and Key Results，目标 + 关键结果（更灵活，鼓励挑战）
- **KPI**：Key Performance Indicator，关键绩效指标（更刚性，考核用）
- **PM 应用**：OKR 定方向（季度），KPI 盯执行（周/月）

### SLA（服务水平协议）
- **解释**：Service Level Agreement，服务提供方对客户的质量承诺
- **核心指标**：
  - **可用率**：99.9%（年宕机 < 8.76 小时）
  - **首次响应时长**：30 分钟内
  - **解决时长**：4 小时内
  - **故障恢复时长**：1 小时内
- **PM 应用**：未达成通常按合同赔偿（扣款、延服务期）；SLA 指标要拆到架构和监控上
- **案例**：企学宝对接的 SLA 承诺——"培训通知 5 分钟触达、成绩同步 1 小时完成"
- **SLO/SLI 补充**：
  - **SLO**（Service Level Objective）：服务水平的内部目标（比 SLA 更严，是 SLA 的安全垫）
  - **SLI**（Service Level Indicator）：服务水平的具体指标（如错误率、延迟）

## 2. 需求分析

### 需求来源
- 用户反馈 / 数据分析 / 竞品分析 / 业务方需求 / 战略规划

### 需求评估维度（4 个）
- **商业价值**：对业务指标的贡献
- **用户价值**：解决用户什么问题
- **技术成本**：开发难度和资源
- **ROI**：投入产出比

### 优先级矩阵

| | 价值高 | 价值低 |
|---|---|---|
| **成本高** | 战略性投入 | 放 backlog |
| **成本低** | 快速做 | 砍掉 |

### 用户画像

格式：`As a ... I want to ... so that ...`

## 6. 文档体系（3 层 + 2 类）

> PM 文档不是越全越好，而是要看"决策层 + 时机"。

### 三层核心文档

| 文档 | 回答什么 | 给谁看 | 写多少 |
|------|---------|--------|--------|
| **BRD**（商业需求文档）| 做这个值不值得 | 老板/决策层 | 5-10 页 |
| **MRD**（市场需求文档）| 市场要什么 | 市场/产品 | 10-20 页 |
| **PRD**（产品需求文档）| 具体怎么做 | 研发/设计 | 20-50 页 |

### 两类辅助文档

| 文档 | 全称 | 用途 |
|------|------|------|
| **FSD** | Functional Specification Document 功能规格文档 | 详细功能技术规格，研发自测用 |
| **UAT** | User Acceptance Testing 用户验收测试 | 用户验收用例，业务方确认 |

### 文档选择时机
- **老板拍板资源**：BRD（算账 + 风险 + ROI）
- **市场/产品对齐**：MRD（市场分析 + 竞品 + 用户需求）
- **研发拆任务**：PRD（功能 + 流程 + 验收）
- **技术实现细节**：FSD（接口 + 数据结构）
- **业务方验收**：UAT（用例 + 通过标准）

### 案例：代理商平台接企学宝
- **BRD**：给老板算"这个对接能减少 XX% 培训成本、提升 XX% 代理活跃度、6 个月回本"
- **MRD**：给市场/产品分析"企学宝是行业头部、目标代理 100% 覆盖、培训内容能直接复用"
- **PRD**：给研发拆解"账号映射接口、标签同步规则、通知推送时序、异常弹窗 5 类"

## 9. 测试与质量保障

### 9.1 冒烟测试（Smoke Testing）

**一句话定义**：保证"软件能起来"的最浅测试——验证核心流程能跑通，不深究细节。**目的不是找 bug，是过滤明显的"开不起来"**。

**关键 3 维**：
- 目的：验证"主流程不挂"——不是找 bug
- 范围：关键路径走一遍，能正常打开/登录/发送——不测异常分支
- 结论：通过→转详细测试；不通过→**当天打回开发，不进详细测试**

### 4 个关键边界（避免被挑战）

- ❌ **不是单元测试**：不针对函数/方法级别
- ❌ **不是集成测试**：不测模块交互
- ❌ **不是验收测试**：不测业务需求满足度（那是 UAT 的事）
- ❌ **不是回归测试**：不验证历史 bug 没复发

> **口诀**：冒烟测试 = "**冒个烟看冒不冒黑烟**"——能正常冒白烟（跑起来）就过，冒黑烟（跑不起来）就返工。

### 5 步 checklist（普渡版本）

1. **核心入口能打开**（登录页 / 主菜单）
2. **主流程走通**（最关键的一笔业务单据走完）
3. **数据能写入**（新建/编辑能保存到 DB）
4. **数据能读出**（列表/详情能正常显示）
5. **最常用按钮响应**（搜索/筛选/导出）

> 5 步全过 = **冒烟通过**，可以进详细测试。任一步卡 = **冒烟不通过**，**当天打回**。

### 普渡场景案例：6/17 企学宝一期上线

**6/12 提测当天，QA 第一步就是冒烟测试**：
1. 飞书扫码登录（企学宝入口）→ 能不能跳？
2. 映射后看到企学宝课程列表 → 能不能显示？
3. 点开一门课 → 能不能播？
4. 考试提交 → 成绩能不能回写到代理商平台？

✅ 4 步都过 → 详细测试（功能、性能、异常）
❌ 任一步卡住 → 6/12 当天打回开发修

### 相邻黑话对比（被问"和 X 什么区别"用得上）

| 概念 | 一句话 | 谁做 | 时机 |
|------|-------|------|------|
| **冒烟测试** | 软件能跑起来 | QA | 提测当天 |
| **回归测试** | 改完代码后没把老功能搞坏 | QA | 每次发版前 |
| **UAT**（用户验收）| 业务方确认"满足需求" | 业务方 | 上线前 |
| **灰度** | 只对部分用户放量 | 运维/PM | 上线后 |

### 提测日标准流程

```
1. 提测邮件/IM 通知（产品/QA/研发）
2. 研发部署到测试环境
3. QA 冒烟测试（5 步 checklist）
   ├─ 通过 → 进入详细测试周期
   └─ 不通过 → 当天打回，循环回到第 1 步
4. 详细测试（功能 / 性能 / 异常 / 兼容性）
5. Bug 修复 + 回归测试
6. UAT（业务方验收）
7. 上线
```

> **关键**：**冒烟测试是"准入门槛"**，不通过就不进详细测试——这能避免"带病进入详细测试"导致的时间和精力浪费。

### 容易踩的坑

- ⚠️ **冒烟通过 ≠ 无 bug**：冒烟只验证"能起来"，详细 bug 留给详细测试
- ⚠️ **环境差异**：测试环境冒烟通过 ≠ 生产环境没问题（数据库、网络、配置都可能不同）
- ⚠️ **数据准备**：冒烟测试前要准备好测试数据（账号、订单、库存等），否则跑不通
- ⚠️ **不要替代 UAT**：冒烟测试是技术验证，UAT 是业务验证，两件事都做

### 记忆口诀

- **"白烟过、黑烟回"**：能跑通就过，跑不通就打回
- **"5 步清单"**：入口→主流程→写→读→按钮
- **"4 个不是"**：不是单测 / 不是集成 / 不是验收 / 不是回归
