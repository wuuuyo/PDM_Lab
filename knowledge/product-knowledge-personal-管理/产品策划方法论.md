# 1. 战略与定位（决定"做对的事"）

## 1.1 北极星指标（North Star Metric）⭐补全项 1

> 补全自:产品经理八股.md 5. 数据分析

**解释**:用一个"最能反映产品核心价值"的数字给整个团队"指北"。

**好北极星的 3 个特征**:
1. **用户从产品获得核心价值时它会拉高** —— 反映"用户真的从中获益"
2. **业务增长时它会拉高** —— 反映"业务真在增长",不是自嗨
3. **团队任何人能解释"我做的事怎么影响它"** —— 不是高管的 KPI,是一线员工也能用的"指南针"

**选错北极星的代价**:KPI 全达成但产品失败。例:把"DAU"当北极星 → 团队为拉 DAU 做签到、弹窗、虚假活动,DAU 涨了但用户没获得价值,留存崩盘。

**案例**:代理商平台北极星候选对比

| 候选 | 问题 |
|------|------|
| DAU | 容易被刷量拉高,不代表真有用 |
| GMV | 涨了但代理可能不开心不续约 |
| AI 咨询次数 | 自娱自乐,没真解决问题 |
| **代理月活解决工单数** ⭐ | 反映"代理觉得平台有用"——最贴近核心价值 |

**PM 应用**:每个季度评审前,先问"我做的这个需求,是拉高北极星、还是只让某个 KPI 好看?"。只拉高 DAU 不拉高工单解决数的功能,要砍掉或重做。

### 指标体系设计（北极星 → 一级指标 → 二级指标）

```
北极星指标(NSM):代理月活解决工单数
    │
    ├─ 一级指标:工单总量 / 工单解决率 / 代理活跃率
    │
    ├─ 二级指标:
    │   ├─ 渠道效率(获客):新代理首月工单数 / 渠道来源占比
    │   ├─ 激活(上手):首日完成工单比例 / AI 悬浮球首次使用率
    │   ├─ 留存(留下):7 日活跃代理 / 30 日活跃代理 / 流失率
    │   ├─ 变现(收入):订阅续费率 / 增值服务使用率
    │   └─ 推荐(传播):代理推荐新代理数 / NPS 净推荐值
    │
    └─ 行动指标(PM 可直接干预):
        ├─ AI 悬浮球首响时长
        ├─ 工单页 P50 / P95 加载时长
        └─ 关键功能使用率
```

**PM 应用**:指标体系是"对北极星的可分解"——北极星是"一句话目标",指标体系是"具体怎么衡量",行动指标是"我下周能改什么"。

## 1.2 商业模式画布（BMC, Business Model Canvas）⭐补全项 2

> 补全自:产品经理八股.md 1. 核心模型
> 来源:daily-log 6/22

**解释**:把商业模式拆成 9 个格子,PM 拿到新方向时先画一遍,看"商业逻辑能不能跑通"。

| # | 格子 | 回答什么 |
|---|------|---------|
| 1 | **客户细分**（CS）| 我服务谁？ |
| 2 | **价值主张**（VP）| 我为他们解决什么问题？ |
| 3 | **渠道通路**（CH）| 我怎么触达他们？ |
| 4 | **客户关系**（CR）| 我怎么和他们互动？ |
| 5 | **收入流**（R$）| 他们为什么付钱？ |
| 6 | **核心资源**（KR）| 我需要什么资源？ |
| 7 | **关键活动**（KA）| 我必须做什么？ |
| 8 | **重要伙伴**（KP）| 谁帮我一起做？ |
| 9 | **成本结构**（C$）| 我要花什么钱？ |

**案例**:代理商平台"AI 悬浮球"的 BMC

| 格子 | 填写 |
|------|------|
| CS | 区域代理商（核心）、客服（次要）|
| VP | 5 秒发起售后咨询,跨系统不用切 |
| CH | 代理商平台首页常驻入口 + 飞书推送 |
| CR | 自助为主 + AI 转人工兜底 |
| R$ | 不直接变现,但提活跃度带动培训订阅 |
| KR | LLM API + 客服知识库 + 行为数据 |
| KA | 对话设计 + 知识库更新 + 模型调优 |
| KP | 企学宝(内容供应)、阿里云(算力)|
| C$ | API 调用费(月 5-10 万) + 维护人力(1 FTE)|

**PM 应用**:**9 格填完如果发现"价值主张"清晰但"收入流"弱,要么补变现路径,要么承认"作为留存工具推进、不能单独立项"**。

## 1.3 价值主张画布（VPC, Value Proposition Canvas）⭐补全项 3

> 补全自:产品经理八股.md 1. 核心模型
> 来源:daily-log 6/30

**解释**:把"产品价值"和"用户需求"对齐的双面图——左侧产品,右侧用户,两侧"对上"=真解决问题,"对不上"=做了没人要。

```
┌──────────────────────────────┐  ┌──────────────────────────────┐
│      产品侧 (Product)         │  │      用户侧 (Customer)        │
├──────────────────────────────┤  ├──────────────────────────────┤
│ 产品/服务 (Products & Services)│  │ 任务 (Jobs to be Done)        │
│                                │  │                                │
│ 痛点解药 (Pain Relievers)     │←→│ 痛点 (Pains)                  │
│                                │  │                                │
│ 收益创造 (Gain Creators)      │←→│ 收益 (Gains)                  │
└──────────────────────────────┘  └──────────────────────────────┘
```

**案例**:AI 悬浮球的 VPC

| 用户侧 | 产品侧 |
|--------|--------|
| **任务**:快速解决售后卡点 | **产品**:AI 对话 + 知识库 |
| **痛点**:跨系统反复切换登录 | **解药**:一句话描述问题,AI 自动派单 |
| **收益**:5 分钟搞定问题 | **创造**:AI 给出 3 个备选方案 |

→ 严丝合缝对得上,真解决了用户问题。

**PM 应用**:PRD 评审前 10 分钟画一遍,**3 个判断口诀**:
- 左侧有产品特性但找不到对应用户痛点 → 砍
- 右侧有用户痛点但左侧找不到产品解药 → 补
- 两侧对得上 → 这版就做这个

## 1.4 OKR（Objectives & Key Results）

> 来源:产品经理八股.md 1. 核心模型

**解释**:目标管理工具。"目标 O"是定性的、鼓舞人心的方向;"关键结果 KR"是 3-5 个可量化的指标,衡量 O 是否达成。

**OKR vs KPI**:

| 维度 | OKR | KPI |
|------|-----|-----|
| 性质 | 灵活、挑战性 | 刚性、考核用 |
| 完成度 | 0.7 就算成功 | 必须 100% 达成 |
| 透明度 | 全员可见 | 通常不公开 |
| 周期 | 季度 | 月/季度 |

**PM 应用**:OKR 帮你"知道自己做的事为什么重要"——把每周做的需求挂到季度 KR 上,周报/复盘里能清晰展示"我做的 AI 悬浮球对话效果,贡献了 KR1 30% 的提升"。

# 2. 用户研究（决定"对的人"）

## 2.1 用户画像（User Persona）

> 来源:产品经理八股.md 3. 用户研究

**解释**:通过调研把目标用户抽象成典型的"虚拟人物",包含基本属性、行为特征、目标动机、痛点障碍。

**4 大要素**:
1. **基本属性**:年龄/职业/地区/收入/教育
2. **行为特征**:使用频率/常用功能/触达渠道
3. **目标动机**:用产品要达成什么
4. **痛点障碍**:为什么没达成/什么阻止了他

**案例**:代理商平台的"区域代理小王"画像

| 维度 | 描述 |
|------|------|
| 基本 | 30 岁,大专,深圳,月入 1.5 万,管 30+ 客户 |
| 行为 | 每天 50+ 工单,习惯用手机查进度 |
| 目标 | 当月完成续约,客户不流失 |
| 痛点 | 跨系统反复切换登录、查一个工单要点 5 次 |

**PM 应用**:做功能时"对齐小王用起来顺不顺"——他看不懂/嫌麻烦的功能,90% 是失败功能。

---

## 2.2 同理心地图（Empathy Map）

> 来源:daily-log 6/30

**解释**:把用户"看不见的内心"画出来——6 个维度:**说/想/做/感受/痛点/目标**。

| 维度 | 区域代理小王（代理商平台）|
|------|--------------------------|
| **说（Says）** | "这培训视频太长了根本看不完" |
| **想（Thinks）** | "我能不能只看考点不学其他的" |
| **做（Does）** | 跳过视频直接搜题库 |
| **感受（Feels）** | 焦虑,怕考不过影响接单 |
| **痛点（Pains）** | 没时间看完 1 小时培训 |
| **目标（Gains）** | 快速通过考试、有资格接单 |

**PM 应用**:基于这张图设计"5 分钟考点速览"功能,比"AI 悬浮球加表情包"这种自嗨功能靠谱 10 倍。

**3 个核心问题**:用户嘴上在抱怨什么？心里真正担心什么？为了目标愿意做出什么妥协？

---

## 2.3 用户旅程地图（User Journey Map）

> 来源:产品经理八股.md 4. 产品设计 / daily-log 6/26

**解释**:把用户从"第一次听说产品"到"成为忠实用户"的整段路拆成阶段,标出**目标/行为/触点/情绪曲线/痛点**。

**5 段式标准结构**:

```
认知 → 考虑 → 首次使用 → 习惯 → 复购/推荐
  ↑     ↑        ↑          ↑        ↑
 触点  触点     触点       触点      触点
 情绪  情绪     情绪       情绪      情绪
 痛点  痛点     痛点       痛点      痛点
```

**案例**:企学宝代理商用户旅程

| 阶段 | 触点 | 情绪 | 痛点 |
|------|------|------|------|
| 认知 | 飞书推送上线公告 | +1 | 没注意到 |
| 考虑 | 点进代理商平台看入口 | 0 | 入口埋太深 |
| 首次使用 | 扫码登录企学宝 | +1 | 扫码失败 |
| 学习 | 看培训视频 | -2 | 视频太长 |
| 考试 | 提交并查成绩 | -1 | 反馈不够细 |
| 习惯 | 每月主动学 | +2 | 通知太频繁 |

**PM 应用**:**情绪低谷段就是优化重点**——"学习"段情绪 -2,优先加"5 分钟考点速览"和"模拟考试"功能。

---

## 2.4 可用性测试（Usability Testing）

> 来源:daily-log 6/29

**解释**:找 5-8 个真实用户、给 3-5 个具体任务、观察他们怎么操作、卡在哪、问什么。**5 个用户就能发现 85% 的可用性问题**(Nielsen Norman Group 经验法则)。

**3 种形式**:
- **Moderated**(测试员边观察边追问):最有价值,适合上线前
- **Unmoderated**(后台录屏+点击流):省人力,但追问不了
- **A/B 远程异步**:大流量产品的灰度验证

**3 步流程**:
1. 写 3-5 个"用户可能卡住"的任务(如"请找'你上个月看的培训视频'")
2. 找 3-5 个**不是同事**的真实用户(同事太熟会给好评)
3. 看他们"卡在哪、问什么、表情如何",改完再上线

**案例**:V17 升级功能上线前可用性测试

| 任务 | 原版通过率 | 改版后 |
|------|----------|--------|
| "请查看你这台机器人的固件版本" | 2/5 卡 2 分钟 | 改名"软件版本",15 秒搞定 |
| "请找到这次升级的内容说明" | 1/5 找到 | 放首页 banner,5/5 找到 |
| "请主动发起升级" | 3/5 担心 | 加"建议升级"绿色徽章 |

**PM 应用**:**可用性测试找的是"用户心智和你的设计不一致"的地方,不是测"功能对不对"**——后者是 QA 的事。

---

## 2.5 用户访谈（User Interview）

**解释**:一对一深度访谈,挖出用户"为什么这样做"。

**5 步结构**:
1. **暖场**（5 分钟）:聊点轻松的,建立信任
2. **行为回顾**（10 分钟）:"上次遇到 XX 问题,你做了什么？"
3. **痛点深挖**（15 分钟）:"那时候最难的是什么？"
4. **方案探索**（10 分钟）:"如果有办法,理想是什么样的？"
5. **总结确认**（5 分钟）:"我理解你刚才说的是...对吗？"

**3 个不要**:
- ❌ 不要问"你想要什么功能" → 用户也不知道自己未来要什么
- ❌ 不要引导性提问 → "你是不是觉得 X 不好？"会带偏
- ❌ 不要在会议室做 → 让用户在自己真实工作环境里操作

**PM 应用**:**访谈前先列 10 个问题,但只背 3 个**——剩下 7 个根据用户的回答即兴追问,才有价值。

# 3. 需求管理（决定"做哪些"）

## 3.1 需求评估 4 维度

> 来源:产品经理八股.md 2. 需求分析

**解释**:PM 拿到一个需求不能拍脑袋说"做"或"不做",要按 4 维度量化。

| 维度 | 量化问题 |
|------|---------|
| **商业价值** | 为业务带来什么？收入/成本/效率各占多少？ |
| **用户价值** | 解决用户什么痛点？多少人/多高频/多痛？ |
| **技术成本** | 研发要多少人力/时间/风险？ |
| **ROI** | 综合下来值不值？ |

**案例**:代理商平台 3 个需求评估

| 需求 | 商业 | 用户 | 技术 | ROI | 决策 |
|------|------|------|------|-----|------|
| AI 悬浮球支持语音输入 | 中 | 高 | 中 | 不错 | ✅ 本 Sprint |
| 工单页加深色模式 | 低 | 中 | 低 | 一般 | ⏸ Backlog |
| 重做整个权限体系 | 看场景 | 低 | 极高 | 差 | 🔪 拆 3 个小需求 |

---

## 3.2 RICE 评分法（精细化需求排序）

> 来源:daily-log 6/23

**解释**:用一个标准公式给每个需求打分排序。

**公式**:`RICE Score = (R × I × C) / E`

| 字母 | 含义 | 取值 |
|------|------|------|
| **R** | Reach 触达多少用户 | 数字（如 500）|
| **I** | Impact 影响多大 | 1=微小 2=低 3=中 4=高 5=巨大 |
| **C** | Confidence 信心指数 | 0-100% |
| **E** | Effort 投入工作量 | 人月 |

**案例**:代理商平台 3 个候选需求

| 需求 | R | I | C | E | Score |
|------|---|---|---|---|-------|
| AI 悬浮球支持语音输入 | 500 | 3 | 80% | 3 | **400** |
| 工单页加深色模式 | 200 | 1 | 50% | 1 | 100 |
| 标签同步对接海外代理 | 80 | 4 | 90% | 4 | 72 |

**PM 应用**:把讨论从"谁拍脑袋"变成"谁有数据"——分数高的先做。

---

## 3.3 MoSCoW 优先级法（轻量级版本分桶）

> 来源:daily-log 6/29

**解释**:用 4 个字母把需求按"必须做"程度分桶。

| 字母 | 含义 | 处理 |
|------|------|------|
| **Must have** | 必须有,没它这版本就不该上 | 必做 |
| **Should have** | 应该有,没它有遗憾但能上 | 优先做 |
| **Could have** | 可以有,时间够就做、没时间就砍 | 争取做 |
| **Won't have** | 这版不做,先放 backlog | 不做 |

**案例**:V17 上线前 1 周确定本版本范围

| 桶 | 需求 |
|----|------|
| Must(4) | OTA 4 集群分时升级 / 失败自动回滚 / 上线公告 / 客户通知模板 |
| Should(3) | 升级进度条 / 失败原因展示 / 客服知识库更新 |
| Could(3) | 升级前自动备份 / 多语言文案 / 升级成功动画 |
| Won't(2) | AI 推荐升级时机 / 客户分级灰度（放 V18）|

**PM 应用**:**RICE + MoSCoW 配合用**——先用 RICE 从 10 个候选里排"分数前 5",再用 MoSCoW 把前 5 个分"M/S/C/W"决定"这版本做哪几个"。

## 3.5 砍需求（PM 的"必杀技"）

> 来源:产品经理八股.md 2. 需求分析（隐含）

**5 个砍需求的理由**:
1. **价值不明确** — "做了也没人用"
2. **ROI 太低** — 投入远超回报
3. **范围蔓延** — 原 PRD 没写,后期加进来
4. **依赖缺失** — 必须等另一个系统先就绪
5. **时机不对** — 这版不合适,下版再说

**砍需求的 3 个话术**:
- "这个我先记下,这版我们聚焦核心场景"（温和）
- "这个 ROI 不够,能不能砍掉 30% 范围变成 1 周能做完？"（折中）
- "这个我们做 P2,先做更重要的"（明确）

**PM 应用**:**砍需求的能力 = 把资源留给更重要的事的能力**——PM 不砍需求,团队会被拖死。

# 4. 设计与交付（决定"做对"）

## 4.1 用户故事（User Story）

> 来源:产品经理八股.md 3. 用户研究

**格式**:`As a [用户角色], I want [做某事], so that [实现某个目标]`

**配套验收标准（Given/When/Then）**:
> 来源:daily-log 6/28

| 字段 | 作用 | 示例 |
|------|------|------|
| **Given** | 前置条件 | 我之前和 AI 聊过 3 轮 |
| **When** | 用户动作 | 我再次点击悬浮球 |
| **Then** | 系统响应 | 页面自动展开最近 1 轮对话摘要 |

**3 条 Given/When/Then = 3 个测试用例**——主路径 1 + 边界状态 1 + 异常分支 1。

**PM 应用**:**每个用户故事至少配 3 条验收标准**——只写"做什么"不写"做到什么程度"的 PRD,研发交付出来大概率"够用但不是你想要的"。

---

## 4.2 信息架构（Information Architecture, IA）

> 来源:产品经理八股.md 4. 产品设计 / daily-log 6/28

**解释**:把"一堆功能/内容"组织成"用户能找到、能理解、能使用"的结构。

**核心 4 件事**:
1. **组织系统** — 功能怎么分组
2. **导航系统** — 用户怎么跳转
3. **标签系统** — 菜单/按钮叫什么
4. **搜索系统** — 找不到怎么办

**画 IA 的 3 步法**:
1. 拿张纸把所有功能/内容画成方块
2. 按"用户找东西的心智路径"重排(不是按你的开发目录)
3. 找 3 个真实用户跑"找任务测试",看他们点哪、卡哪

**案例**:代理商平台 IA 改造效果

| | 改造前 | 改造后 |
|---|---|---|
| 分组 | 按业务模块 | 按"用户角色 + 高频任务" |
| 培训入口点击率 | 12% | **38%** |

**PM 应用**:**IA 决定"产品好不好用"的 80%**,再炫的视觉救不回来混乱的结构。

## 4.3 PRD 标准 7 章节

> 来源:产品经理八股.md 4. 产品设计 + 工作流程.md

```
1. 背景与目的
   - 业务背景
   - 需求目的
   - 目标用户

2. 用户故事
   - As a / I want / so that

3. 功能需求
   - 功能点 1/2/3
   - 业务流程
   - 验收标准(Given/When/Then)

4. 非功能需求
   - 性能要求
   - 安全要求
   - 兼容性

5. 风险与依赖
   - 风险点
   - 依赖方

6. 排期计划
   - 计划开始
   - 计划结束
   - 负责人

7. 验收标准
   - UAT 用例
   - 上线标准
```

## 4.4 PRD 评审 Checklist（7 维自检）⭐补全项 4

> 补全自:产品经理八股.md 7. 项目管理
> 来源:daily-log 6/25

**PRD 写完 → 评审会前 15 分钟,自检 7 维**:

| # | 维度 | 自检问题 |
|---|------|---------|
| 1 | 背景/目的 | 一句话能说清"为什么做"吗？ |
| 2 | 用户故事 | 主要场景都有故事覆盖吗？ |
| 3 | 功能边界 | 写明"做什么"和"不做什么"了吗？ |
| 4 | 业务流程 | 流程图画了吗？ |
| 5 | 异常分支 | 异常状态列了吗？（如"用户没登录""接口超时"）|
| 6 | 验收标准 | 每个功能点都 Given/When/Then 了吗？ |
| 7 | 依赖/排期/风险 | 列了吗？负责人明确吗？ |

**反着用**——评审会上研发提"这块没写",PM 翻 checklist 知道"对,这块是第 5 维我马上补",而不是"哦我忘了"。

**案例**:弹窗 PRD 漏掉"用户没登录态"分支(4 类异常 → 实际 5 类),就是没跑 checklist 里的"异常分支"——上线后才发现要打回重做。

# 5. 数据与增长（决定"做得好"）

## 5.1 AARRR（海盗模型）

> 来源:产品经理八股.md 1. 核心模型

**5 阶段深度拆解**:

| 阶段 | 核心指标 | 优化动作 |
|------|---------|---------|
| **Acquisition** 获客 | CAC、渠道来源 | 渠道质量分析、SEO/SEM |
| **Activation** 激活 | 首日完成核心动作率 | 首次体验优化、新手引导 |
| **Retention** 留存 | 次日/7 日/30 日留存 | 召回机制、价值交付 |
| **Revenue** 变现 | ARPU、付费转化 | 定价策略、套餐设计 |
| **Referral** 推荐 | 邀请转化率、K 因子 | 分享激励、口碑传播 |

**PM 应用**:每一步都有独立指标,**不能只盯"获客"不看"留存"**。

---

## 5.2 上瘾模型（Hook Model）

> 来源:daily-log 6/27

**4 步循环**:触发 → 行动 → 多变的奖励 → 投入

| 步 | 含义 | 抖音案例 |
|----|------|---------|
| 触发 | 外部推送/内部情绪 | 推送通知/无聊 |
| 行动 | 阻力极小的操作 | 打开 App |
| 多变的奖励 | 不知道下次会刷到什么 | 下一条视频的未知性 |
| 投入 | 用户产生数据/内容 | 点赞、关注、收藏 |

**PM 应用**:做产品时反问自己"用户为什么每天回来"——靠"必须用业务"驱动 vs 靠"想用就用"驱动,设计策略完全不同。

---

## 5.3 Cohort 队列分析（同期群分析）

> 来源:daily-log 6/27

**解释**:把用户**按"首次使用时间"分组**,看**每组在后续时间的行为**(7 日/30 日留存、复购率),而不是"所有用户平均"。

**"平均数陷阱"案例**:企学宝上线后,整体 7 日留存 30%。但 Cohort 一拆——6/17 首登的代理 7 日留存 45%,6/22 首登的代理 7 日留存只有 18%。

**PM 应用**:**所有"留存问题"先跑 Cohort 再说结论**——平均数会掩盖真相。

---

## 5.4 NPS 净推荐值

> 来源:daily-log 6/27

**公式**:`NPS = 推荐者%(9-10分) - 贬损者%(0-6分)`(范围 -100 到 +100)

**判断标准**:
- **> 50** 优秀
- **> 30** 良好
- **< 0** 危险

**PM 应用**:**NPS 的真正价值不是数字本身,是"贬损者的具体吐槽"**——深度追问后分类汇总,高频痛点就是下一期 P0。

---

## 5.5 A/B 测试 5 步

> 来源:产品经理八股.md 5. 数据分析 / daily-log 6/20

| 步 | 关键动作 |
|----|---------|
| 1 | **定义假设** — "X 改版后转化率会提升 10%" |
| 2 | **设计实验** — 分流比例、观察时长、核心指标 |
| 3 | **分配流量** — 新老用户各 50%、灰度放出 |
| 4 | **分析结果** — 数据有显著差异才作数(p<0.05) |
| 5 | **决策上线** — 胜出版全量、败出版回滚 |

**PM 应用**:**指标必须"先于上线"定义好**——不然跑完数据你也不知道"涨了"是因为按钮位置还是别的因素。

---

## 5.6 增长飞轮（Growth Flywheel）

> 来源:补全(daily-log 6/26 内串联)

**解释**:找到"用户增长 → 产品更好 → 用户更愿意来"的正反馈循环,而不是线性漏斗。

**案例**:代理商平台"代理活跃度"飞轮

```
代理活跃使用
    ↓
积累数据 + 改进产品
    ↓
产品更好用
    ↓
新代理主动注册
    ↓
代理活跃使用(循环)
```

**PM 应用**:**找到你的飞轮,把每个动作挂上去**——每个新需求反问"它推动飞轮的哪一环"。

# 6. 协作与决策（决定"和谁一起做"）

## 6.1 RACI 矩阵

> 来源:daily-log 6/26

**4 种角色**:

| 角色 | 含义 | 数量 |
|------|------|------|
| **R** Responsible | 执行人,真干活 | 可多个 |
| **A** Accountable | 责任人,兜底拍板 | **只能 1 个** |
| **C** Consulted | 咨询人,给意见 | 可多个 |
| **I** Informed | 知会人,结果通知 | 可多个 |

**案例**:V17 OTA 升级的 RACI(涉及 6 个部门)

| 部门 | 角色 |
|------|------|
| 研发 | R(出固件包) |
| 运维 | R(监控升级状态) |
| **产品** | **A(出问题产品兜底)** |
| 客服 | C(看上线公告提供反馈) |
| 销售 | I(升级期间不发新单) |
| 供应链 | I(主板版本变更知会) |

**3 步落地**:
1. 列出事项
2. 标 R/A/C/I
3. 找"A 超过 1 个"或"全是 R 没 A"的格子重调

**PM 应用**:**RACI 跑完,谁也不甩锅**——研发知道自己得干、产品知道自己兜底、客服知道自己该给客户怎么答。

---

## 6.2 鱼骨图（Ishikawa Diagram）

> 来源:daily-log 6/26

**6M 分类**:Man(人) / Machine(机器) / Method(方法) / Material(物料) / Measurement(测量) / Environment(环境)

**案例**:V17 升级"国内集群升级成功率只有 96.5%"

| 分类 | 可能原因 |
|------|---------|
| Man | 运维值班操作漏步骤 |
| Machine | OTA 服务器在德国机房,国内访问慢 |
| Method | 升级窗口 22:00 但部分代理时区偏早 |
| Material | 新固件包 1.2GB,部分弱网下载失败 |
| Measurement | 监控告警阈值太高,问题没暴露 |
| Environment | 部分代理餐厅 WiFi 不稳定 |

**PM 应用**:**鱼骨图的核心不是"画得多漂亮"而是"团队一起画"**——研发看技术、客服看用户、运维看流程,3 视角拼出全貌。

---

## 6.3 5Why 分析法

> 来源:daily-log 6/23

**解释**:遇到问题不要停在"表面原因",连续追问 5 次"为什么"找到"根本原因"。

**案例**:配送机器人"突然停在餐厅中央不动"

| Why | 答案 |
|-----|------|
| Why 1 | 激光雷达报警 |
| Why 2 | 检测到正前方 30cm 有障碍 |
| Why 3 | 顾客把椅子推到机器人正前方 |
| Why 4 | 机器人避障策略是"绕行",但卡在桌椅间窄通道没法绕 |
| Why 5 | 建图时这块区域宽度只够机器人单行通过,没在地图上标记"单行通道"约束 |
| **根因** | **地图数据缺失"单行通道"标记** |
| **解决方案** | 建图阶段让服务员标注单行通道(不是优化避障算法)|

**PM 应用**:**不 5Why 的话,可能去优化避障算法,治标不治本**。

---

## 6.4 PDCA 戴明环

> 来源:daily-log 6/23

**4 步循环**:Plan(计划) → Do(执行) → Check(检查) → Act(处理)

**核心思想**:**螺旋上升**——每跑一轮 PDCA 改进一个具体问题,再跑下一轮再改进一个。

**案例**:代理商平台"工单平均处理时长"优化

| 轮 | Plan | Do | Check | Act |
|----|------|----|----|-----|
| 1 | P50 4h → 2.5h | 改版工单页 + AI 推荐 | 3h(达预期 80%) | AI 推荐用得少,改规则引擎 |
| 2 | 2.5h → 2.3h | 规则引擎 + 反馈学习 | 2.3h ✅ | 进入下一轮优化 |

**PM 应用**:**复盘是"事后总结经验",PDCA 是"边做边改的循环"**——强调"持续"。

---

## 6.5 设计冲刺（Design Sprint）

> 来源:daily-log 6/30

**Google Ventures 的 5 天产品决策法**:

| 天 | 做什么 | 产出 |
|----|--------|------|
| 周一 | 理解 | 和专家聊、定问题 |
| 周二 | 发散 | 画草图不评论 |
| 周三 | 决策 | 投票选方案 |
| 周四 | 原型 | 做最简可用原型 |
| 周五 | 测试 | 5 个真用户跑一遍 |

**核心思想**:**5 天节奏逼出所有假设**——不是真要做 5 天。

**案例**:V17 升级前跑过一轮设计冲刺——周一对齐"升级失败要自动回滚还是人工介入"、周二画 3 套方案、周三投票(自动回滚胜出)、周四做纸面原型、周五找 5 个代理测试。**结果**:3/5 代理问"如果回滚失败会怎样"——PRD 之前没写这个问题,最后补进 PRD。

**PM 应用**:重要功能上线前、需求大但方向不明时跑一轮设计冲刺,哪怕压缩成 5 小时也能帮 PM **① 在动手前暴露分歧 ② 用真用户验证而不是"领导拍板" ③ 节省"做完才发现方向错"的几周返工**。

---

## 6.6 影响地图（Impact Mapping）

**解释**:用"目标 → 参与者 → 影响 → 交付物"4 层结构,把"为什么做"和"做什么"对齐。

**4 层结构**:

```
目标 (Goal)
  ↓ 服务谁
参与者 (Actor)
  ↓ 怎么影响他们
影响 (Impact)
  ↓ 怎么实现
交付物 (Deliverable)
```

**案例**:代理商平台"降低工单解决时长"

| 层 | 内容 |
|----|------|
| 目标 | 区域代理工单 P50 解决时长 4h → 2.5h |
| 参与者 | 区域代理、客服 |
| 影响 | 代理能更快速找到答案、客服能复用历史方案 |
| 交付物 | AI 悬浮球 + 工单页改版 + 历史方案库 |

**PM 应用**:**让团队对齐"为什么做"**——避免"做完发现不是用户要的"。

---

## 6.7 复盘 8 步法

> 来源:工作流程.md 复盘 SOP

**8 步**:
1. 确认范围(time-context 拿日期)
2. 拉日历(本周所有会议)
3. 拉妙记(4 份/周典型)
4. 拉群消息(关键对接群)
5. 读工作日志(还原日常工作)
6. 结构化分析(11 字段)
7. 写复盘表(NO.XXX)
8. 关联需求(link 字段挂本周需求)

**PM 应用**:**8 步缺一步,复盘就会变成"印象流"**,失去"可索引、可追溯、可复盘经验"的价值。

# 7. 跨部门沟通 4 原则 ⭐补全项 5

> 补全自:产品经理八股.md 8. 沟通协作 + daily-log 6/24

**4 原则**:

1. **明确共同目标** — 双方对齐的"北极星"
2. **用数据说话** — 别只说"提升体验",要给具体数字
3. **提前对齐预期** — 不要等上线才说"做完了"
4. **及时同步进展** — 进展/卡点/变更都要主动通气

**案例**:企学宝对接方案评审时,研发 Lead 一上来就问"做这个 KPI 怎么算",你掏出"活跃代理提升 30%、培训完成率从 25% 涨到 60%、Webhook 接口 P99 < 1 秒"三个具体数字,研发马上能评估工作量、给排期。

**反例**:"让代理用得更爽" → "自嗨"层面对话。
