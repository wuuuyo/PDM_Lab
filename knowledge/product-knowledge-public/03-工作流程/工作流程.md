# 工作流程

> 🔄 **分类**：五、工作流程（需求处理 + 文档规范）
> 日常工作中沉淀的流程和规范——以"可复用"为标准沉淀

---

## 需求处理 7 步流程

### 1. 需求接收
- **来源**：用户反馈 / 业务方 / 数据分析 / 竞品
- **动作**：进入需求池，标注来源和初步描述

### 2. 需求分析
- 理解业务背景和目标
- 明确目标用户和使用场景
- 梳理核心流程和边界
- 评估价值和优先级

### 3. 方案设计
- 撰写 PRD
- 制作原型/低保真
- 确定验收标准

### 4. 评审对齐
- 内部评审（PM 组）
- 跨部门评审（研发/设计/测试）
- 确认排期和分工

### 5. 开发跟踪
- 加入需求看板
- 跟踪开发进展
- 处理技术疑问
- 验收功能实现

### 6. 上线发布
- 确认上线时间
- 准备上线文档
- 跟踪上线效果

### 7. 复盘总结
- 回顾目标达成
- 记录问题和解决方案
- 更新知识库

## 文档规范

### PRD 模板（7 章节标准结构）

```
# [需求名称]

## 1. 背景与目的
- 业务背景
- 需求目的
- 目标用户

## 2. 用户故事
- 作为...我希望...以便...

## 3. 功能需求
### 3.1 功能点 1
- 描述
- 业务流程
- 验收标准

### 3.2 功能点 2
...

## 4. 非功能需求
- 性能要求
- 安全要求
- 兼容性

## 5. 风险与依赖
- 风险点
- 依赖方

## 6. 排期计划
- 计划开始
- 计划结束
- 负责人
```

### PRD 写作 4 原则
1. **结构清晰**：7 章节固定，方便跨人查阅
2. **需求可测**：每个功能点都有验收标准
3. **边界明确**：说清楚"做什么"和"不做什么"
4. **风险可视**：列清楚依赖方 + 风险点

## 与各部门协作 SOP

### 与研发协作
- **对接时机**：需求评审前 1 天发 PRD 预告
- **对接方式**：技术方案评审会
- **响应时效**：技术疑问 24 小时内回复
- **协作工具**：飞书项目 / 多维表格

### 与设计协作
- **对接时机**：PRD 评审通过后
- **对接方式**：设计评审会 + 走查
- **响应时效**：原型 3-5 个工作日
- **协作工具**：Figma / 即时设计

### 与测试协作
- **对接时机**：开发完成前 1 天
- **对接方式**：测试用例评审会
- **响应时效**：测试用例 1 个工作日
- **协作工具**：测试用例库

### 与业务方协作
- **对接时机**：需求启动前对齐 + 上线前验收
- **对接方式**：需求宣讲会 + 验收会
- **响应时效**：业务方反馈 1 个工作日
- **协作工具**：飞书群 + 邮件

## 复盘 SOP（W23 立规，W24 落地）

### 默认节奏
- **每周五 18:00** 推送复盘报告（USER.md 立规）
- **例外**：重大节点冲刺/上线前，可临时提前到周四，**需在 W 复盘文件里标注**（MEMORY 立规 6/11）

### 8 步走完

1. **确认范围** — `time-context` 拿到本周日期范围（不要凭记忆）
2. **拉日历** — `calendar +agenda` 拉本周所有日历事件，输出 9 个会议清单
3. **拉妙记** — `vc +notes --meeting-ids <id>` 按会议 ID 拉智能纪要（4 份/周典型），整理关键决策和行动项
4. **拉群消息** — `im +chat-messages-list` 拉关键群聊（对接群/方案沟通群/项目群）本周消息
5. **读工作日志** — 读 `memory/work/daily/{本周日期}.md` 还原日常工作
6. **结构化分析** — 按 11 字段组织内容（见下方"复盘表字段"）
7. **写复盘表** — 在「个人需求管理系统」Base 的「复盘表」（table_id=`tblbeEeBpcBsLfaL`）按 NO.XXX 编号写入
8. **关联需求** — 通过 link 字段关联本周涉及的需求 record_id（W23 关联 5 个，W24 关联 3 个）

### 复盘表 11 字段（个人需求管理系统 → 复盘表）

| 字段 | 类型 | 用途 |
|------|------|------|
| ID | auto_number | NO.001 自动递增 |
| 复盘周期 | text | "2026-06-08 ~ 2026-06-14" |
| 本周完成 | text | 已完成的任务/需求（多行）|
| 关键进展 | text | 里程碑/评审通过/对齐完成 |
| 进行中事项 | text | 跨周推进的项目 |
| 关键卡点 | text | 问题+卡在哪+截止+负责人 |
| 未考虑到的地方 | text | 遗漏点+原因+教训 |
| 沉淀与收获 | text | 方法论/经验教训 |
| 下周计划 | text | 优先级最高的事项 |
| 同类问题解决方案 | text | 下次遇到同类怎么解决 |
| 关联需求 | link → 需求表 | 本周涉及的需求 record_id 列表 |

### 复盘表写入命令模板

```bash
# 1. 准备 batch JSON
jq '{fields: ["复盘周期","本周完成",...,"关联需求"], rows: [[<value1>,<value2>,...]]}' \
   /home/gem/.aily/workdir/w{N}_review.json \
   > /home/gem/.aily/workdir/w{N}_review_batch.json

# 2. 批量创建
larksuite-cli base +record-batch-create --as user \
  --base-token "QTrqbJ0wDadiZasNii5ctD61nle" \
  --table-id "tblbeEeBpcBsLfaL" \
  --json "$(cat /home/gem/.aily/workdir/w{N}_review_batch.json)"
```

### 复盘产物 3 件套

- **复盘表**（NO.XXX）：结构化记录，可索引、可关联
- **周复盘文件** `memory/work/weekly/2026-W{N}.md`：详细版，含链接、引用
- **IM 推送摘要**：用 `im_message` 推给用户，3 行总结 + 复盘表链接

### 关联需求识别

- W23 关联的 5 个需求 record_id：`recviZSHCzeWNq`（企学宝）/ `recvln3jRpRClX`（飞书消息推送）/ `recXQNm2ZGlg61`（CRM）/ `recvln3jjA4QEM`（企学宝调研）/ `recgSI6zYHaIoh`（巡航任务）
- 新增需求需先在「需求表」（table_id=`tbl4ZqjX8c8OyQhG`）创建，再在复盘表 link 字段引用
- 复盘表写完后**必须验证关联**：打开 Base 链接，肉眼检查「关联需求」字段是否显示
