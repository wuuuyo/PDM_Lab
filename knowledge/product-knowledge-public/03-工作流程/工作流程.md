# 工作流程

> 🔄 **分类**：五、工作流程（需求处理 + 文档规范）
> 日常工作中沉淀的流程和规范——以"可复用"为标准沉淀

---

## 需求处理 7 步流程

### 1. 需求接收
- **来源**：用户反馈 / 业务方 / 数据分析 / 竞品
- **动作**：进入需求池，标注来源和初步描述

### 2. 需求分析
- 理解业务背景和目标
- 明确目标用户和使用场景
- 梳理核心流程和边界
- 评估价值和优先级

### 3. 方案设计
- 撰写 PRD
- 制作原型/低保真
- 确定验收标准

### 4. 评审对齐
- 内部评审（PM 组）
- 跨部门评审（研发/设计/测试）
- 确认排期和分工

### 5. 开发跟踪
- 加入需求看板
- 跟踪开发进展
- 处理技术疑问
- 验收功能实现

### 6. 上线发布
- 确认上线时间
- 准备上线文档
- 跟踪上线效果

### 7. 复盘总结
- 回顾目标达成
- 记录问题和解决方案
- 更新知识库

## 文档规范

### PRD 模板（7 章节标准结构）

```
# [需求名称]

## 1. 背景与目的
- 业务背景
- 需求目的
- 目标用户

## 2. 用户故事
- 作为...我希望...以便...

## 3. 功能需求
### 3.1 功能点 1
- 描述
- 业务流程
- 验收标准

### 3.2 功能点 2
...

## 4. 非功能需求
- 性能要求
- 安全要求
- 兼容性

## 5. 风险与依赖
- 风险点
- 依赖方

## 6. 排期计划
- 计划开始
- 计划结束
- 负责人
```

### PRD 写作 4 原则
1. **结构清晰**：7 章节固定，方便跨人查阅
2. **需求可测**：每个功能点都有验收标准
3. **边界明确**：说清楚"做什么"和"不做什么"
4. **风险可视**：列清楚依赖方 + 风险点

## 与各部门协作 SOP

### 与研发协作
- **对接时机**：需求评审前 1 天发 PRD 预告
- **对接方式**：技术方案评审会
- **响应时效**：技术疑问 24 小时内回复
- **协作工具**：飞书项目 / 多维表格

### 与设计协作
- **对接时机**：PRD 评审通过后
- **对接方式**：设计评审会 + 走查
- **响应时效**：原型 3-5 个工作日
- **协作工具**：Figma / 即时设计

### 与测试协作
- **对接时机**：开发完成前 1 天
- **对接方式**：测试用例评审会
- **响应时效**：测试用例 1 个工作日
- **协作工具**：测试用例库

### 与业务方协作
- **对接时机**：需求启动前对齐 + 上线前验收
- **对接方式**：需求宣讲会 + 验收会
- **响应时效**：业务方反馈 1 个工作日
- **协作工具**：飞书群 + 邮件

## 知识库管理 SOP

### 日常积累
- 每天 10:00 自动推送 3 个 PM 知识点到 `daily-log.md`
- 每周五 18:00 复盘——把值得长期保留的知识点从 `daily-log.md` 迁移到对应主题文件

### 月度整理
- 把 `daily-log.md` 中高频出现的术语沉淀到 `行业通用词语.md`
- 把高频需求类型沉淀到本文档的"典型需求场景"章节

### 知识检索
- **遇到陌生术语** → 查 `行业通用词语.md`
- **写 PRD 前** → 查 `工作流程.md`（本文档）+ `产品经理八股.md`
- **跨部门沟通前** → 查 `产品经理八股.md` 的"沟通协作"
