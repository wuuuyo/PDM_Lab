## 第一部分：技术架构

### 1.1 云服务模式（4 种 X as a Service）

> 记忆口诀：**IaaS 卖算力 / PaaS 卖平台 / SaaS 卖软件 / HaaS 卖硬件**

| 词语 | 全称 | 解释 | 案例 |
|------|------|------|------|
| **IaaS** | Infrastructure as a Service 基础设施即服务 | 卖"服务器/存储/网络"等底层资源 | AWS EC2、阿里云 ECS |
| **PaaS** | Platform as a Service 平台即服务 | 卖"开发平台/中间件"，让开发者专注业务 | Heroku、阿里云函数计算 |
| **SaaS** | Software as a Service 软件即服务 | 卖"成品软件订阅"，客户按月/年付费，厂商统一部署升级 | 飞书、钉钉、Salesforce、企学宝 |
| **HaaS** | Hardware as a Service 硬件即服务 | 卖"硬件订阅"，客户按月付使用费，所有权不转移，运维打包 | 配送机器人按月租、医疗 CT 按扫描次数付、打印机按张数付 |

#### 本公司的"云服务"地图

| 类型 | [本公司]对应 | 说明 |
|------|---------|------|
| **IaaS** | 阿里云、华为云 | 别人做的，本公司租用 |
| **PaaS** | 阿里云函数计算 | 别人做的 |
| **SaaS（外部）** | 企学宝、飞书、Salesforce | 本公司订阅使用 |
| **SaaS（自研）** | X 培训平台（bizlearnify.com）| [本公司]自己做的多租户 SaaS |
| **HaaS（未来）** | 配送机器人按月租 | 未来方向，订阅制 + 服务打包 |
| **本地软件** | 代理商平台、运营管理平台 | [本公司]自研，未来可 SaaS 化 |

### 1.2 SaaS 关键指标

| 指标 | 全称 | 含义 |
|------|------|------|
| **ARR** | Annual Recurring Revenue | 年度经常性收入 |
| **MRR** | Monthly Recurring Revenue | 月度经常性收入 |
| **Churn Rate** | 流失率 | 客户停用比例 |
| **LTV** | Life-Time Value | 客户生命周期价值 |
| **CAC** | Customer Acquisition Cost | 客户获取成本 |
| **NRR** | Net Revenue Retention | 净收入留存率（>100% 才是好 SaaS）|

### 1.3 SaaS PM 特有设计点

- **多租户**：一套服务服务多客户，租户间数据隔离
- **订阅计费**：按月/年/用量/超额付费（不是一次性买断）
- **API 开放**：客户要集成，必须开放
- **租户级配置**：皮肤、品牌、字段可按租户配置（不写死）
- **向后兼容**：升级不能影响老客户
- **SLA 承诺**：99.9% 可用率是常见的，要写入合同
- **数据导出**：客户要能带走数据（防锁定）

### 1.4 多租户（Multi-Tenancy）3 种模式

| 模式 | 隔离级别 | 成本 | 适合场景 |
|------|---------|------|---------|
| 共享数据库 + tenant_id 区分 | 低 | 低 | 中小型 SaaS |
| 共享数据库 + 独立 schema | 中 | 中 | 中型 SaaS |
| 独立数据库 | 高 | 高 | 大客户/金融/医疗 |

> X 培训平台用的是**子站点 + 共享数据库**模式（每个子站点 = 一个租户），4 个子站点隔离。

### 1.5 产品标准化术语

| 词语 | 全称 | 解释 | 案例 |
|------|------|------|------|
| **SPU** | Standard Product Unit 标准化产品单元 | 产品的"概念集合"，描述"这是一类什么产品" | [本公司]"欢乐送"是 1 个 SPU；iPhone 16 Pro 是 1 个 SPU |
| **SKU** | Stock Keeping Unit 库存计量单元 | 产品的"具体一箱/一台"，带具体属性 | 欢乐送-标准版/欢乐送-Pro-带顶视-白色 = 2 个 SKU |
| **BOM** | Bill of Materials 物料清单 | 产品所需的全部零部件及数量 | 机器人 BOM：主板+激光雷达+轮组+... |
| **ECN** | Engineering Change Notice 工程变更通知 | 产品设计/工艺的变更通知 | "这款机器人主板 V2.0 发出 ECN" |

#### SPU vs SKU 关键差异

> **SPU = 产品的"型号"（粗颗粒度）** — 跨代升级
> **SKU = 产品的"具体一箱/一台"（细颗粒度，带属性）** — 同代内属性变化

| 维度 | SPU | SKU |
|------|-----|-----|
| 颗粒度 | 粗 | 细 |
| 变化频率 | 季度/年级 | 一次发布会新增几十个 |
| 颗粒度增长 | 加法 | **乘法**（颜色×尺码×容量 = 大量 SKU）|
| 用途 | 营销/规划 | 仓储/销售/库存 |

#### PM 设计原则

- **SPU 控制在 5-15 个**，单 SPU 下 SKU 控制在 20-50 个
- 选 3-5 个关键属性（颜色、尺寸、容量、版本），不要 10 个
- SKU 4 个生命周期：新品期 / 主销期 / 滞销期 / 退市期
- SPU/SKU/库存/订单/财务 5 套数据**实时同步**（否则对账灾难）

## 第二部分：业务管理

### 2.1 ERP（Enterprise Resource Planning 企业资源计划）

**定义**：整合企业核心业务（财务/采购/库存/生产/销售/人力）的管理系统，让数据一处录入、处处可用。

**关键特征**：
- 🏢 企业级：服务整个公司
- 🔗 高度集成：财务+采购+库存+生产+销售+人力**一套系统**
- 💾 统一数据库：所有模块共享数据源
- 🔄 流程自动化：跨部门流程自动触发
- 📊 实时报表：业务数据实时汇总

**经典 ERP 8 大模块**

> 💡 记忆口诀：**「财采销存、生人客供」**

| 模块 | 管什么 | [本公司]场景类比 |
|------|--------|-------------|
| **财务** | 总账、应收应付、固定资产、报表 | [本公司]财务系统 |
| **采购** | 采购申请、订单、收货、付款 | 物料采购 |
| **销售** | 报价、订单、发货、回款 | 代理商订单 |
| **库存** | 入库、出库、盘点、调拨 | **运营管理平台-调拨维保** |
| **生产** | BOM、工单、生产排程、车间管理 | 机器人装配线 |
| **人力** | 员工档案、考勤、薪资、绩效 | HR 系统 |
| **CRM** | 客户、商机、合同、售后 | 代理商/客户管理 |
| **供应链** | 物流、仓储、供应商 | 物流系统 |

### 2.2 CRM（Customer Relationship Management 客户关系管理）

- 管理企业与客户关系的系统和方法
- 案例：代理商平台对接 CRM，实现客户信息统一管理

### 2.3 SCM（Supply Chain Management 供应链管理）

- 机器人配件和维修备件管理

### 2.4 OA（Office Automation 办公自动化）

- 流程审批、文档管理、协同办公
- [本公司]用飞书审批 + OA 调拨单

### 2.5 IPD（Integrated Product Development 集成产品开发）

- 一套产品开发流程和方法论，强调跨部门协作、异步开发、结构化流程
- 案例：华为引入 IPD 后，产品上市周期大幅缩短

### 2.6 BU（Business Unit 业务单元）

- 企业内部相对独立的业务部门/利润中心
- 案例："这个需求需要协调多个 BU"

### 2.7 服务请求类

| 词语 | 全称 | 解释 | 案例 |
|------|------|------|------|
| **SR** | Service Request 服务请求 | 客户服务请求，通常指售后问题处理 | "收到一个 SR，需要处理机器故障" |
| **PR** | Problem Report / Pull Request | 问题报告 / 代码合并请求 | 研发场景指代码合并；客服场景指问题报告 |

## 第四部分：权限与安全

### 4.1 权限安全的核心框架

权限安全不是“给用户配几个菜单”这么简单，而是 To B 系统的基础可信能力。PM 可以用一句话理解：**先确认你是谁，再判断你能做什么，最后留下你做过什么的证据**。

| 层级 | 解决的问题 | PM 需要关注什么 | 常见产物 |
|------|------------|----------------|----------|
| **认证 Authentication** | 你是谁？ | 登录方式、账号体系、身份可信度、异常登录提醒 | 登录页、SSO、MFA、账号绑定 |
| **授权 Authorization** | 你能做什么？ | 角色、功能权限、数据范围、审批边界 | RBAC、权限矩阵、菜单/按钮控制 |
| **审计 Audit** | 你做过什么？ | 操作留痕、敏感操作记录、可追溯性 | 操作日志、审计报表、导出记录 |
| **数据保护 Data Protection** | 数据是否安全？ | 敏感字段、脱敏、加密、下载/导出限制 | 字段脱敏、加密存储、水印、访问日志 |
| **隔离 Isolation** | 不同组织是否互不影响？ | 租户边界、组织层级、跨组织访问风险 | tenant_id、组织树、数据隔离策略 |

### 4.2 高频术语

| 词语 | 全称 | 解释 | 案例 |
|------|------|------|------|
| **SSO** | Single Sign-On 单点登录 | 用户只需登录一次，即可访问所有相互信任的系统 | 代理商用飞书账号登录代理商平台后，跳转企学宝/CRM 时无需重新登录 |
| **OAuth 2.0** | Open Authorization 2.0 开放授权协议 | 允许第三方系统在用户授权后访问部分资源，而不是直接拿到用户密码 | 用飞书授权登录代理商平台，平台拿到授权 token 后读取基础身份信息 |
| **Token** | 访问令牌 | 登录成功后系统发给客户端的临时凭证，用来证明“你已经通过认证” | 用户登录后请求工单列表时带上 token，后端校验通过才返回数据 |
| **JWT** | JSON Web Token | 一种常见 token 格式，里面可以携带用户 ID、角色、过期时间等信息 | 代理商平台在 JWT 中记录 user_id 和 role，后端根据角色判断能否访问接口 |
| **MFA** | Multi-Factor Authentication 多因素认证 | 除密码外，再用短信、邮箱、验证码器等方式二次确认身份 | 超级管理员登录后台时，需要密码 + 手机验证码，降低账号被盗风险 |
| **账号映射** | Account Mapping | 不同系统间账号的对应关系，确保用户在一个系统登录后，其他关联系统能识别同一用户 | 代理商平台账号与企学宝账号的映射：国内用 SSO+手机号，海外独立对接 |
| **RBAC** | Role-Based Access Control 基于角色的访问控制 | 按角色分配权限，用户属于某个角色，角色拥有一组权限 | 代理商平台的权限体系：管理员拥有全部功能权限，普通客服仅有查看工单权限 |
| **功能权限** | 能否操作某个功能 | 决定用户能看到/操作哪些菜单、按钮 | 代理商能看到"工单管理"菜单，普通客服看不到 |
| **数据权限** | 能否看到某些数据 | 决定用户能看到哪些具体数据范围 | 区域 A 代理商只能看到自己区域的工单，看不到区域 B 的 |
| **最小权限原则** | Principle of Least Privilege | 用户只获得完成工作所需的最少权限，避免“默认全开” | 客服只能查看和回复自己负责区域的工单，不能导出全部客户数据 |
| **操作日志** | Operation Log | 记录用户关键操作，方便追责、排查和审计 | 谁在什么时候修改了客户手机号、导出了哪批订单、删除了哪条配置都要留痕 |
| **敏感字段脱敏** | Data Masking | 对手机号、身份证、邮箱、地址等敏感信息进行部分隐藏 | 客服列表只显示 138****1234，只有授权角色才能查看完整手机号 |
| **字段加密** | Field Encryption | 对敏感字段加密存储，即使数据库泄露也难以直接读取明文 | 客户证件号、合同金额、支付信息等字段在数据库中加密保存 |
| **多租户隔离** | Multi-tenant Isolation | 多个客户/组织共用一套系统时，数据和配置必须按租户隔离 | A 代理商登录后只能看到自己组织的客户、订单和培训记录，不能访问 B 代理商数据 |

### 4.3 PM 写权限需求时要问的 6 个问题

1. **谁能进系统？**支持账号密码、SSO、手机号，还是第三方 OAuth？
2. **有哪些角色？**超级管理员、组织管理员、普通成员、只读用户是否边界清楚？
3. **能操作哪些功能？**菜单、按钮、接口是否都要受控，前端隐藏不等于后端安全。
4. **能看到哪些数据？**按组织、区域、部门、本人、客户归属，还是项目归属过滤？
5. **哪些行为要留痕？**新增、编辑、删除、导出、审批、授权、登录失败都可能需要日志。
6. **异常情况怎么处理？**离职账号、角色变更、跨组织协作、权限缓存、token 过期是否有规则？

### 4.4 权限矩阵示例

| 角色 | 功能权限 | 数据权限 | 敏感操作 | 适用场景 |
|------|----------|----------|----------|----------|
| **超级管理员** | 全部菜单与配置 | 全组织数据 | 可授权、导出、删除 | 平台运营方或公司管理员 |
| **组织管理员** | 组织内成员、客户、订单、工单管理 | 本组织数据 | 可新增成员、分配角色 | 代理商老板或区域负责人 |
| **客服/运营** | 工单查看、回复、跟进 | 本人或本区域工单 | 不可导出全量数据 | 日常服务处理 |
| **只读用户** | 查看报表、查看记录 | 被授权的数据范围 | 不可编辑、删除、导出 | 财务、管理层、审计人员 |

### 4.5 常见坑

- **只做前端隐藏，不做后端校验**：用户看不到按钮，但直接调接口仍可能越权。
- **角色太多且没有命名规则**：后期没人敢改权限，系统变成“祖传配置”。
- **功能权限和数据权限混在一起**：能点“工单管理”不代表能看全部工单。
- **导出权限被低估**：导出一次就是批量数据泄露风险，要单独控制和留痕。
- **离职/转岗没有回收机制**：账号还在、token 还有效，是典型安全隐患。
- **多租户只靠前端传 tenant_id**：租户隔离必须以后端身份和服务端校验为准。
