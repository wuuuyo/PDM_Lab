# 系统架构

> 🏗️ **分类**：二、技术架构（深度版）
> PM 看懂技术架构的"最小必要知识"——不写代码，但要能听懂研发在聊什么、判断需求能不能做、动一处会牵动哪几处
> 持续积累中——按"分层→组件→模式→机器人特有→PM 视角"五段式沉淀

---

## 1. 架构基础（PM 必须懂的核心概念）

### 1.1 三层架构（最经典的分层思维）

> 所有系统都可以拆成 3 层，**PM 提任何需求都先问"动哪一层"**

| 层级 | 职责 | 技术组件举例 | 普渡对应 |
|------|------|------------|---------|
| **表现层** | 用户看到和操作的界面 | Web（React/Vue）、移动端（iOS/Android/小程序）、桌面端 | 代理商平台 Web、运营管理平台 Web |
| **业务层** | 处理业务逻辑、流程编排、规则校验 | 后端服务（Java/Go/Node）、API 网关 | 订单服务、工单服务、CRM 服务 |
| **数据层** | 存储、查询、数据分析 | 关系数据库（MySQL/PG）、缓存（Redis）、搜索（ES）、数仓 | 代理商平台 DB、培训记录库 |

**PM 应用**：写需求时明确"动哪几层"。例："工单页加深色模式" = 只动**表现层**（前端 CSS）；"接企学宝同步培训记录" = 动**业务层 + 数据层**（要新加同步服务、新加表）。分层意识能让 PM 评估"这个需求改 1 天还是 1 个月"。

### 1.2 单体架构 vs 微服务架构

| 维度 | 单体（Monolith） | 微服务（Microservices） |
|------|------------------|------------------------|
| **形态** | 所有功能打包在一个应用里 | 按业务域拆成几十个独立小服务 |
| **部署** | 一处改全部重新部署 | 每个服务独立部署、独立伸缩 |
| **技术栈** | 一种语言/框架 | 多种语言共存 |
| **团队协作** | 改动容易冲突 | 每个团队负责一个服务 |
| **故障影响** | 一个 bug 全站挂 | 一个服务挂了不影响其他 |
| **适合** | 产品初期（用户<10万）| 规模化后（用户>百万、服务多）|

> **普渡现状**：代理商平台 + 运营管理平台是**模块化单体**（按业务模块拆代码包，但部署还是一个应用），还没到微服务阶段。

**PM 应用**：**不要在单体阶段硬上微服务**——成本高、收益小。**触发微服务化的 3 个信号**：① 单次部署超过 30 分钟 ② 一个团队改动影响另一个团队的发布 ③ 不同模块的资源消耗差 10 倍以上。

### 1.3 前后端分离

**传统模式**：后端渲染 HTML 页面（PHP/JSP），前端写死在 HTML 里
**现代模式**：前端是独立 SPA（单页应用），后端只提供 JSON API

| 优势 | 挑战 |
|------|------|
| 前端后端并行开发，效率高 | SEO 不友好（要 SSR 解决）|
| 一套后端 API 服务多端（Web/iOS/Android）| 跨域问题（CORS）|
| 前端技术栈灵活（React/Vue）| 接口管理成本上升（要 OpenAPI/GraphQL）|

**普渡**：代理商平台是前后端分离架构，前端是 Vue、后端是 Java。

## 2. 关键组件（PM 经常听研发提到的"零件"）

### 2.1 数据库（数据层核心）

| 类型 | 用途 | 普渡场景 |
|------|------|---------|
| **关系数据库**（MySQL/PG）| 结构化数据，强事务 | 代理商平台、工单、订单 |
| **缓存**（Redis/Memcached）| 高频读、临时数据 | Session、Token、热数据 |
| **搜索引擎**（Elasticsearch）| 全文搜索、复杂查询 | 工单搜索、知识库检索 |
| **时序数据库**（InfluxDB/TDengine）| 时间序列数据 | 机器人运行数据、传感器日志 |
| **对象存储**（OSS/S3）| 大文件存储 | OTA 固件包、培训视频 |
| **消息队列**（Kafka/RocketMQ）| 异步任务、削峰填谷 | 通知推送、数据同步 |

**PM 应用**：**写需求必问"数据存在哪"**——涉及"数据搬家"的需求（接企学宝、接 CRM）一定涉及数据层改造。

### 2.2 中间件（连接各个服务的"基础设施"）

| 中间件 | 干什么 | PM 视角的翻译 |
|--------|-------|--------------|
| **API 网关** | 统一入口、限流、鉴权 | "所有外部调用都从这进" |
| **消息队列** | 异步任务、削峰填谷 | "用户点完通知后，后台慢慢处理、不卡页面" |
| **配置中心** | 动态改配置不重启 | "改个功能开关不用发版" |
| **服务注册/发现** | 服务之间怎么找到对方 | "微服务之间互相打电话要拨号本" |
| **链路追踪** | 一次请求经过哪些服务 | "用户报'工单页卡了'，研发能定位到是哪个服务慢" |

### 2.3 缓存策略（最常被问的"性能优化"）

> **PM 必懂 3 个问题**：
> ① 什么数据要缓存？答：**读多写少**的（用户档案、字典、配置）
> ② 缓存多久？答：**按业务容忍度**（5 分钟？1 小时？1 天？）
> ③ 缓存和数据库不一致怎么办？答：**最终一致** + 设置合理的过期时间

**经典 3 模式**：
- **Cache-Aside**（旁路缓存）：应用先查缓存，命中就返回；不命中查 DB 再写缓存。**最常用**
- **Write-Through**（写穿透）：写 DB 时同步写缓存。**强一致场景**
- **Write-Behind**（写回）：先写缓存，异步写 DB。**性能高但可能丢数据**

## 3. 架构模式（成熟方案，直接套）

### 3.1 多租户架构（SaaS 化必备）

> 之前在 `行业通用词语.md` 1.4 节讲过 3 种隔离模式，这里补充**选型决策树**

```
是否在金融/医疗等强合规行业？
├─ 是 → 独立数据库（合规+客户定制）
└─ 否 → 是否需要租户级字段/表结构定制？
    ├─ 是 → 共享 DB + 独立 schema
    └─ 否 → 共享 DB + tenant_id（普渡云学堂当前方案）
```

**普渡场景**：
- **普渡云学堂**：共享 DB + tenant_id，4 个子站 = 4 个租户
- **代理商平台**：单租户（只服务普渡自己），但内部有"代理商 → 客户"的二级权限隔离
- **未来 SaaS 化**：代理商平台如果对外开放给其他机器人厂商，就要升级到多租户

### 3.2 读写分离（数据库性能优化标配）

**模式**：主库写、从库读，应用层按 SQL 类型路由

| 适用 | 不适用 |
|------|--------|
| 读多写少（10:1 以上）| 写多读少（写主库直接挂）|
| 单表数据量 > 1000 万行 | 表小、业务简单 |
| 报表/分析类查询 | 强实时一致（金融交易）|

**普渡场景**：代理商平台工单列表查询是典型读多写少，配置了 1 主 2 从；订单表写多读少，没做读写分离。

### 3.3 分库分表（数据库扛不住时的最后一招）

| 维度 | 垂直拆分 | 水平拆分 |
|------|---------|---------|
| **做法** | 按业务把表分到不同库 | 按某个字段（user_id/order_id）把同一张表分到多库 |
| **典型场景** | 用户库、订单库、商品库分开放 | 订单表按月份分 12 个库 |
| **PM 视角** | "业务大了，一个 DB 装不下" | "单表 1 亿行查询太慢" |

**触发条件**（PM 该知道的红线）：
- 单表 > **5000 万行** 或 单库 > **2TB**
- 单条 SQL 走全表扫描 > **3 秒**
- 数据库 CPU 长期 > **70%**

**普渡场景**：当前代理商平台单表最大也就几百万行，**还没到分库分表阶段**。PM 知道有这根红线即可。

## 4. 高可用与高并发（PM 必懂的"系统健壮性"）

### 4.1 3 大经典问题

| 问题 | 解释 | 解决方案 |
|------|------|---------|
| **高可用**（HA）| 系统挂的时间少 | 主备切换、集群部署 |
| **高并发** | 同时很多人访问不卡 | 缓存、限流、扩容 |
| **高扩展** | 加机器就能扛更多流量 | 微服务、读写分离、CDN |

**PM 应用**：写非功能需求时，**"能扛多少 QPS/多少用户"** 一定要明确——研发不会主动告诉你系统能扛什么。

### 4.2 限流 / 降级 / 熔断（保护系统的"三道闸"）

| 机制 | 作用 | 触发场景 |
|------|------|---------|
| **限流** | 限制每秒最多 N 个请求 | 秒杀活动、API 被刷 |
| **降级** | 主动关掉非核心功能 | 大促关推荐、保核心 |
| **熔断** | 下游服务挂了，上游自动断开 | 企学宝接口超时 → 代理商平台停止调用 |

**普渡场景**：企学宝对接必须设计熔断——如果企学宝接口挂了 30 秒，代理商平台不能卡死，要自动"切回老逻辑"或"提示用户稍后重试"。

### 4.3 监控告警（"线上出问题能第一时间发现"）

> **3 大监控支柱**：
> ① **Metrics**（指标）：QPS、延迟、错误率、CPU 内存
> ② **Logs**（日志）：请求链路、错误堆栈
> ③ **Traces**（链路追踪）：一次请求经过哪些服务、各花了多少时间

**PM 应用**：上线后必问"监控接好了吗"——没监控就是"盲飞"。

## 5. 安全架构（PM 容易忽略的"红线"）

### 5.1 认证 vs 授权

| 概念 | 回答什么 | 普渡组件 |
|------|---------|---------|
| **认证**（Authentication）| 你是谁？ | 飞书 OAuth、用户名密码 |
| **授权**（Authorization）| 你能做什么？ | RBAC 角色权限、数据权限 |
| **审计**（Audit）| 你做了什么？ | 操作日志、登录日志 |

### 5.2 常见攻击与防御（PM 知道术语即可）

| 攻击 | 防御 |
|------|------|
| **SQL 注入** | 参数化查询、ORM |
| **XSS** | 输入过滤、输出编码 |
| **CSRF** | Token 验证、SameSite Cookie |
| **DDoS** | CDN、高防 IP、限流 |
| **数据泄露** | 字段加密、脱敏、最小权限 |

**PM 应用**：**To B 系统的数据安全是"必答题"**——尤其涉及客户档案、培训记录、订单数据。PRD 里要明确"敏感字段加密"、"权限最小化"、"操作留痕"。

## 6. 云原生与运维（PM 知道就行）

| 概念 | 解释 | 普渡使用情况 |
|------|------|------------|
| **容器化**（Docker）| 应用打包成镜像，"一处构建处处运行" | 普渡后端服务已容器化 |
| **K8s** | 容器编排系统，"自动部署 + 自动伸缩" | 部分核心服务在 K8s |
| **CI/CD** | 持续集成/持续部署，"代码提交后自动测试自动上" | 普渡已有 CI/CD 流水线 |
| **IaC** | 基础设施即代码，"服务器配置写成脚本可追溯" | 部分使用 Terraform |
| **Serverless** | "不用管服务器，写函数就行" | 还未使用 |

**PM 应用**：写非功能需求时——"环境一致性"（本地跑通 ≠ 测试环境跑通 ≠ 生产跑通）需要 CI/CD 保证；"弹性伸缩"是云原生的核心价值。

## 7. 机器人/IoT 系统特有架构（普渡专属）

### 7.1 云-边-端架构

```
┌──────────┐         ┌──────────┐         ┌──────────┐
│   云端    │  4G/WiFi │  边缘端   │  局域网  │   设备端  │
│ (普渡云) │ ←────→  │(机器人本体)│ ←────→ │(传感器)  │
└──────────┘         └──────────┘         └──────────┘
   ↑                                              ↑
   业务平台                                   实时控制
   (代理商平台)                               (低延迟要求)
   - 用户管理                                 - SLAM
   - 订单/工单                                - 避障
   - 数据分析                                 - 运动控制
   - 报表                                     - 紧急制动
```

| 层级 | 延迟要求 | 普渡组件 |
|------|---------|---------|
| **设备端** | < 10ms | 实时避障、紧急制动 |
| **边缘端**（机器人本体）| < 100ms | SLAM、路径规划 |
| **云端** | < 1s | 业务平台、报表、OTA |

**PM 应用**：**实时性需求必须明确放哪一层**——"机器人撞到障碍物要立刻停下"是设备端的事，不能依赖云端。

### 7.2 OTA 空中升级架构

```
代理商平台(运营管理)
       ↓ 生成升级包
CDN 分发
       ↓ 下载
机器人(边缘端)
       ↓ 校验 + 备份旧版
       ↓ 应用新版
       ↓ 失败回滚
       ↓ 上报结果
代理商平台(状态监控)
```

**PM 视角 3 个关键点**：
- **灰度发布**：先 5% → 20% → 50% → 100%（每档观察 1-2 天）
- **失败回滚**：升级失败 30 秒内自动回到旧版（蓝绿部署）
- **升级窗口**：默认 4 集群分时（国内/新加坡 22:00，德国次日 14:00，美国次日 11:00）

### 7.3 机器人数据流（时序数据 + 业务数据双轨）

| 数据类型 | 来源 | 存储 | 用途 |
|---------|------|------|------|
| **时序数据** | 机器人传感器、运行状态 | 时序 DB（InfluxDB/TDengine）| 实时监控、故障诊断 |
| **业务数据** | 订单、工单、CRM | 关系 DB | 业务流转 |
| **地图数据** | SLAM 建图 | 专用地图服务 | 导航 |
| **视频/图像** | 摄像头、深度相机 | 对象存储 | 巡检、清洁效果检测 |

**PM 应用**：**"机器人数据存储"是个跨多存储的复杂架构**——PM 提需求时要知道"清洁报告里的地图图片从对象存储拉"。

## 8. 架构演进路径（PM 视角的"技术债"认知）

### 8.1 系统演进的 5 个阶段

```
阶段 1：单体 (Monolith)
   ↓ 用户>10万
阶段 2：模块化单体
   ↓ 团队>50人
阶段 3：微服务
   ↓ 流量>100万QPS
阶段 4：服务网格（Service Mesh）
   ↓ 全球化部署
阶段 5：云原生 + Serverless
```

**普渡现状**：
- 代理商平台：阶段 1-2（模块化单体）
- 普渡云：阶段 2-3（部分服务已拆）
- 机器人本体：阶段 2（边缘端轻量化服务）

### 8.2 技术债识别（PM 怎么判断"系统是不是要重构"）

> **技术债 = 现在偷的懒 = 未来要还的债**

**4 大信号**：
1. **改一个需求要 3 倍时间**（"这功能本来 1 天能做完，现在要 3 天"）
2. **每次发版都有"连锁反应"**（"我改 A 模块，B 模块崩了"）
3. **新人看不懂代码**（"老员工不解释就改不动"）
4. **线上事故频发**（"每周都有 P 级故障"）

**PM 应用**：当技术债信号出现时，**主动提"技术债偿还 Sprint"**——不然业务会越来越慢。

## 9. 架构图绘制（PM 必会的"沟通工具"）

### 9.1 C4 模型（4 个层次的架构图）

| 层级 | 回答什么 | 受众 |
|------|---------|------|
| **Context**（系统上下文）| 系统在更大的世界里和谁互动 | 所有人 |
| **Container**（容器）| 系统由哪些"大块"组成（Web/DB/Service）| 研发、PM |
| **Component**（组件）| 每个容器内部由哪些模块组成 | 研发 |
| **Code**（代码）| 具体的类、函数 | 研发 |

**PM 必会画前 2 层**——评审会上能用一张 Context 图让所有人"对齐"。

### 9.2 普渡云系统 Context 图（示例）

```
                     普渡云（代理商平台）

   (代理商)              (销售/客服)            (企学宝/CRM)
```

## 10. PM 必会的 10 个架构问题

> 评审会上被研发问"这个需求怎么做"时，能答出这些 = 入门

| # | 问题 | 答得不专业的反面例子 |
|---|------|---------------------|
| 1 | **数据存在哪？** | "就存起来啊" |
| 2 | **同步还是异步？** | "都行" |
| 3 | **能扛多少 QPS？** | "应该挺多的" |
| 4 | **延迟要求是？** | "快点就行" |
| 5 | **一致性要求？** | "别出错" |
| 6 | **触发频率？** | "用户点就触发" |
| 7 | **失败怎么兜底？** | "应该不会失败" |
| 8 | **数据量预估？** | "几千条吧" |
| 9 | **谁来维护？** | "开发吧" |
| 10 | **上线标准？** | "做完就上" |

**正确答案模板**：
> 1. 数据存在 MySQL 主库 + Redis 缓存
> 2. 异步推送（Kafka 消息队列）
> 3. 预估 QPS 1000（早高峰 + 晚高峰）
> 4. 同步接口 P99 < 200ms，异步任务 < 5min
> 5. 最终一致即可（允许 1 分钟内同步）
> 6. 用户主动触发 + 定时（每天 9:00）
> 7. 接口失败重试 3 次 + 兜底页提示
> 8. 日均新增 1 万条，3 年保留
> 9. 研发 + 运维共同维护（oncall 轮值）
> 10. 灰度 1% → 10% → 100%，监控告警接好
